# UI Rules

## Термины

- Верхний уровень: “Продуктовые направления”.
- Верхние элементы направления: “Продуктовые вехи”.
- Нижний уровень внутри задачи: “Контрольные точки”.
- Не использовать “контур” как системный термин интерфейса. Исключение: если слово является частью осмысленного описания проекта.

## Главная страница

- `index.html` — входная точка для GitHub Pages.
- Общие дизайн-токены держать в `assets/shared.css`; уникальную компоновку страницы держать в page-specific CSS.
- Новые изменения главной сначала вносить в `assets/home.css` и `assets/home.js`.
- Основные действия должны быть собраны в рабочем центре.
- Архив недельных отчетов должен быть компактным и выдерживать 52+ отчета в год.
- Опубликованные отчеты должны визуально отличаться от локальных.
- У опубликованных отчетов не должно быть кнопки “Удалить”.
- В портфельной таблице недельного отчета готовность показывать компактным круговым индикатором, а действие держать узкой иконкой без текстового заголовка.
- Локальные отчеты показываются только для недель, которых еще нет в публичном `reports.json`.
- Для опубликованных отчетов показывать дату отчета, а не “обновлено” из локального браузера.
- Кнопки открытия доски отчетов должны сохранять текущие видимые подписи, но вести в самый свежий доступный отчет, а не в базовый шаблон доски.
- Не дублировать одинаковые CTA в разных местах страницы без необходимости.

## Доска отчета

- Новые изменения в стилях и поведении доски сначала вносить в `assets/weekly-workboard.css` и `assets/weekly-workboard.js`.
- Не копировать одну и ту же правку вручную в опубликованные HTML-отчеты, если ее можно решить через общий CSS/JS.
- На desktop сохранять широкий executive-dashboard.
- На маленьких экранах использовать карточки вместо перегруженных таблиц.
- На телефоне избегать горизонтальной прокрутки.
- Служебные действия на мобильном сворачивать в кнопку “Действия”.
- В рабочем режиме ответственного показывать полный набор действий.
- В опубликованном режиме скрывать админские действия: публикацию, импорт patch, принятие изменений, редактирование JSON/заголовка/направлений и добавление задач.
- В опубликованном режиме оставлять “Сохранить HTML” как аварийный способ передать текущий результат сотрудника.

## Продуктовые направления

- Кнопка “Показать состав” раскрывает состав направления и не должна открывать редактор.
- Двойной клик по строке направления открывает редактирование направления.
- В счетчике показывать формат `Y в работе / X всего`.

## Проекты и задачи

- В карточках задач показывать название, статус, релиз, готовность, сделанное за неделю и контрольные точки.
- Длинные статусы не должны вылезать за границы блока.
- `discovery` отображается как “Анализ”.
- `test` отображается как “Тест”.

## Roadmap

- Новые изменения опубликованных дорожных карт сначала вносить в `assets/fw-release-roadmap.css` и `assets/fw-release-roadmap.js`.
- Новые изменения конструктора дорожных карт сначала вносить в `assets/roadmap-builder.css` и `assets/roadmap-builder.js`.
- `demo-page-main/demo-page-main` можно использовать как read-only источник финальных файлов, но правки делать только в рабочем корне.
- Не брать файлы из `old`, `old/Git` или `backups` как рабочую основу без отдельного решения.
- Опубликованные roadmap должны сохранять компактный desktop-режим для ноутбуков: при ширине до 1440px или высоте до 820px использовать одну линию карточек, а не высокую шахматную раскладку.
- Метка “Сегодня” спокойная синяя, не красная.
- Вертикальная линия “Сегодня” идет из центра плашки.
- На мобильном roadmap может превращаться в вертикальную последовательность контрольных точек.

## Изменения и patch

- Измененные задачи получают бейдж “Есть изменения”.
- Измененные поля подсвечиваются мягко, без агрессивного цвета.
- “Показать изменения” открывает список отличий.
- “Скачать изменения” выгружает только patch.
- “Импортировать изменения” применяет patch точечно.
- “Принять изменения” утверждает текущий отчет как новую базу и убирает подсветку.
- “Сбросить изменения” возвращает отчет к состоянию на момент открытия.

## Скриншоты

- Режим скрина должен убирать служебный шум, но не скрывать смысловые данные.
- Карточка проекта должна оставаться удобной для ручного скриншота.
- PNG-кнопки не должны визуально спорить с основным интерфейсом.
