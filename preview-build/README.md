# Weekly Workboard Healbe

Автономная HTML-платформа для недельных продуктовых отчетов Healbe, дорожных карт и публикации отчетов через GitHub Pages без backend.

## Структура проекта

- `demo-page-main/demo-page-main` — скачанная копия текущего GitHub-состояния. Это read-only источник финальных файлов с сервера: можно читать и копировать в рабочий корень, нельзя менять внутри этой папки.
- Корень проекта — рабочая зона для новых правок перед публикацией. Актуальные исправления сначала складываются сюда.
- `weekly-reports` в корне — рабочая папка недельных отчетов и реестра перед загрузкой.
- `old` — исторические версии, эксперименты и резервные HTML.
- `backups` — снимки состояния перед ревью или крупными изменениями.
- `assets` — общие стили, скрипты, изображения и документы.

- `assets/roadmap-builder.js` — общая логика конструктора дорожных карт и его HTML-экспорта.

## Переход на общую архитектуру

Старые опубликованные HTML-отчеты остаются как архив уже прошедших недель.

Новая логика собирается постепенно:

1. Общие стили и JS выносятся в `assets`.
2. `assets/shared.css` подключается первым и содержит базовые токены, шрифт, box-sizing и общую основу форм.
3. Главная страница подключает `assets/home.css` и `assets/home.js`.
4. Рабочая доска `weekly-reports/weekly_workboard.html` подключает `assets/weekly-workboard.css` и `assets/weekly-workboard.js`.
5. Конструктор дорожных карт `roadmap_builder.html` подключает `assets/roadmap-builder.css` и `assets/roadmap-builder.js`.
6. Опубликованные дорожные карты `site.html`, `release_app.html`, `fw_release_roadmap.html`, `heart_roadmap.html` и `neem_roadmap.html` подключают общие `assets/fw-release-roadmap.css` и `assets/fw-release-roadmap.js`.
7. Единый просмотрщик `weekly-reports/report.html` загружает данные недели из `weekly-reports/data/*.json`.
8. В `reports.json` для новых отчетов можно указывать `report.html?data=<file>.json`, а старые HTML остаются архивом.

Так изменения в дизайне, кнопках и логике можно будет делать в одном месте, а не переносить вручную в каждый новый отчет.

## Публичные и локальные отчеты

Главная страница показывает два источника:

- опубликованные отчеты из `weekly-reports/reports.json`;
- локальные отчеты из `localStorage` текущего браузера.

Опубликованные отчеты имеют приоритет над локальными дублями той же недели. Если неделя уже есть в `reports.json`, главная должна показывать ее как опубликованный отчет и не показывать кнопку удаления локальной копии.

Дата из `reports.json` считается датой отчета, а не временем локального обновления. Локальное “обновлено” показывается только для отчетов, которые еще не опубликованы.

## Публикация новой недели

Новая схема публикует отчет как JSON-данные плюс запись в реестре. Старый HTML-экспорт остается резервом, чтобы можно было откатиться без паники и лишнего героизма.

Кнопка “Подготовить публикацию” в недельной доске готовит новую схему: скачивает JSON-файл данных отчета и обновленный `reports.json`. Кнопка “Сохранить HTML” остается резервным HTML-экспортом.

1. JSON-данные отчета в `weekly-reports/data`;
2. обновленный `weekly-reports/reports.json` с записью об этом отчете.
3. общий `weekly-reports/report.html` и `assets/weekly-workboard.*`, если они менялись.

Минимальная запись в `reports.json`:

```json
{
  "title": "Недельный срез продуктовых работ Healbe",
  "meta": "Неделя №22 | 28 мая 2026",
  "url": "report.html?data=week-22_2026-05-28.json",
  "updatedAt": "2026-05-28",
  "taskCount": 21
}
```

В `url` можно указывать новый просмотрщик `report.html?data=<file>.json` или старый HTML-файл внутри `weekly-reports`.

## Режимы недельного отчета

`weekly-reports/weekly_workboard.html?report=<id>` — рабочий режим ответственного. В нем доступны подготовка публикации, импорт patch, принятие изменений, редактирование JSON, направлений и добавление задач.

`weekly-reports/report.html?data=<file>.json` — опубликованный режим для сотрудников. В нем скрыты админские действия, но доступны просмотр, редактирование существующих задач, “Показать изменения”, “Скачать изменения”, “Сбросить изменения”, PNG и “Сохранить HTML”.

HTML-сохранение в опубликованном режиме оставлено как аварийный канал: если patch по какой-то причине не подходит, сотрудник может скачать весь HTML и передать текущий результат ответственному.

## Черновики дорожных карт

Главная создает локальные черновики дорожных карт в `localStorage` и открывает их через `roadmap_builder.html?draft=<id>`.

Конструктор дорожных карт теперь использует общие файлы из `assets`. При экспорте HTML в файл сохраняются данные конкретной карты, а стили и логика остаются подключены через `assets/roadmap-builder.css` и `assets/roadmap-builder.js`.

## Основной сценарий

1. Ответственный открывает `index.html`.
2. Создает новый отчет недели.
3. Система открывает `weekly-reports/weekly_workboard.html?report=<id>`.
4. Ответственный вносит базовые правки и экспортирует публикационный пакет.
5. JSON-файл отчета и обновленный `reports.json` загружаются на GitHub.
6. Сотрудники открывают опубликованный отчет, редактируют свои задачи и скачивают patch через “Скачать изменения”.
7. Ответственный импортирует patch-файлы через “Импортировать изменения”.
8. После проверки нажимает “Принять изменения”, чтобы сделать текущую версию новой базой.
9. Финальная версия снова экспортируется и публикуется.

## Данные задачи

Основные поля `task`:

- `id` — уникальный идентификатор.
- `domain` — домен или продуктовая зона.
- `owner` — ответственный.
- `title` — название задачи.
- `startDate` — дата старта.
- `releaseDate` — дата релиза.
- `releaseNumber` — ручной номер релиза для release-band.
- `releaseProgress` — общая готовность.
- `weeklyProgress` — движение недели.
- `status` — статус задачи.
- `actualHours` — факт часов.
- `plannedHours` — план часов.
- `description` — краткое описание проекта.
- `artifactTitle` — заголовок недельного артефакта.
- `artifactNote` — текст “Сделано” за неделю.
- `achievements` — контрольные точки.
- `focus` и `ceoFocus` — риск или зона внимания.
- `contourId` / `directionId` — ручная привязка к продуктовому направлению.

## Продуктовые направления

Верхний уровень интерфейса называется “Продуктовые направления”.

Актуальные направления:

- Управление весом.
- Кардио-функции.
- NEEM и устройства.
- РУ.
- B2B и исследования.
- GoBe Lora.
- GoBe U Simple.
- GoBe5.

Если у задачи есть `contourId` или `directionId`, задача привязывается к указанному направлению. Если поля нет, направление определяется автоматически по названию и домену.

## Статусы

В JSON статусы остаются английскими, в интерфейсе показываются русские подписи:

- `planned` — В плане.
- `evaluation` — Оценка.
- `preparation` — Оценка, совместимость со старым значением.
- `discovery` — Анализ.
- `contract` — Договор.
- `in-progress` — В работе.
- `development` — Разработка.
- `test` — Тест.
- `risk` — Риск.
- `paused` — Пауза.
- `done` — Завершено.

Активными для счетчика “Продуктовые вехи” считаются `discovery`, `contract`, `in-progress`, `development`, `test`, `risk`.

## Patch-файлы

Patch-файл содержит только отличия от исходного состояния отчета:

- тип `healbe-weekly-report-patch`;
- `reportId`, `reportTitle`, `reportMeta`;
- `baseVersion`, `baseHash`, `sourceHash`;
- `author`, если пользователь указал имя;
- `createdAt`;
- список измененных задач и полей;
- краткую сводку.

При импорте patch:

- `reportId` должен совпадать;
- при отличающемся `baseHash` показывается предупреждение;
- если текущее значение совпадает с `before`, изменение применяется автоматически;
- если текущее значение уже отличается, открывается окно конфликта;
- конфликт можно оставить как есть, принять из patch, объединить вручную или пропустить.

## Проверка перед публикацией

- Открыть `index.html`.
- Проверить, что опубликованные отчеты из `weekly-reports/reports.json` видны на главной.
- Создать новый отчет недели.
- Проверить, что открылся `weekly-reports/weekly_workboard.html?report=<id>`.
- Изменить поле “Сделано” в одной задаче.
- Открыть “Показать изменения”.
- Скачать patch.
- Нажать “Принять изменения”.
- Убедиться, что подсветка исчезла.
- Нажать “Подготовить публикацию” и проверить скачивание JSON-файла отчета и `reports.json`.
- Проверить опубликованную страницу с cache-bust, например `index.html?v=check`.
